## プロダクト構成（Fusion OS）

このリポジトリは Fusion OS 各システムの静的サイト・アプリ（HTML/JS、Cloudflare Workers 上で動作）をまとめたものです。ファイル名の接頭辞がそのままシステムを表します。

| システム名 | 役割 | 主なファイル例 |
|---|---|---|
| 凪（nagi） | 組織診断エンジン | `nagi_*.html`, `brand_nagi.html` |
| 奏（kanade） | 適材適所・チームビルディング | `kanade_*.html`, `brand_kanade.html` |
| ミツメル（mitsumeru） | 個人ジャーナリング・日次監査（アプリ販売予定） | `mitsumeru_*.html`, `brand_mitsumeru.html` |
| 響（hibiki） | 現場改善・収益向上 | `hibiki_*.html`, `brand_hibiki.html` |
| 弦（yuzuru） | 地域連携・営業 | `yuzuru_*.html`, `brand_yuzuru.html` |

`worker.js` / `hibiki_worker.js` / `wrangler.toml` が Cloudflare Workers のエントリポイントと設定。`index*.html` 群はハブ・ポータル系ページ。

## 開発・レポートの原則

- 完了報告には、変更したファイルパスとコミット（またはPR）を明記する。根拠のない「完了しました」は完了とみなさない。
- 「確認してください」で終わらせない。判断が必要な点は選択肢（原則3つまで、Yes/Noで答えられる形）を添えて提案する。
- 言い訳や謝罪より先に改善策を出す。
- ハルシネーション厳禁。わからないこと・できないことは正直に「わかりません／できません」と言う。実際のコード・コミット履歴・hubページの表示順など、事実は必ず現物を確認してから発言する（過去のコミットログにも「実コードを根拠に再訂正」という反省が複数回残っている）。
- 手作業を増やす提案はしない。できる範囲は基本的にAI（Code）側で完結させる。

## 文体・トーン（サイト上のコピー・文言を書く／直す場合）

- 語尾は「です・ます」調。
- 精神論を排した知的な語り口。エレガントに毒を吐くくらいは良いが、攻撃的にはならない。
- 断定はするが、根拠のない決めつけはしない。
- 福祉・介護現場の空気を知っている人間の言葉を意識する（`hibiki_kaigo.html` / `hibiki_shogai.html` など現場向けページで特に重要）。
